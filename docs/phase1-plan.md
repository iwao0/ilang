# ilang フェーズ1: 四則演算インタプリタ (実装時のプラン)

> このドキュメントは実装時のプラン記録。フェーズ完了後にスコープ拡張を検討するときの出発点として使う。

## Context

ilang は新しいプログラミング言語。最終的なゴールは:

- ライブラリ/クラスごとに `net`, `file` などの**実行権限 (capability)** を持たせ、サプライチェーン攻撃に耐性をもつ
- メモリ安全性は **ARC**
- 関数宣言・型名は **Rust** 風の構文
- 四則演算規則は **C/JavaScript** とほぼ同一

フェーズ1 ではこのうち「数値の四則演算が評価できる最小処理系」のみ実装する。capability・ARC・型システムは未着手。後続フェーズの土台となるよう、ワークスペース構成は最初から複数 crate に分割した。

## 決定事項

| 項目 | 決定 |
| --- | --- |
| 実装言語 | Rust (1.95.0 で実装・テスト) |
| 数値型 | `i64` と `f64` |
| 実行モデル | ツリーウォーク型インタプリタ |
| CLI | REPL (`ilang`) + ファイル実行 (`ilang run foo.il`) |
| `i64` オーバーフロー | `RuntimeError::Overflow` を返す (Rust 流、要再検討) |

## スコープ

- 整数リテラル: `42`
- 浮動小数リテラル: `3.14`, `2.5e-3`
- 二項演算子: `+`, `-`, `*`, `/`, `%`
- 単項演算子: `-`, `+`
- 括弧
- 演算子優先順位・結合性は C/JS と同一
- `i64` と `f64` の混合は `f64` に昇格
- `i64 / 0`, `i64 % 0` はランタイムエラー、`f64` は IEEE754 (`inf`/`NaN`)

**スコープ外**: 変数, `let`, 関数定義, 文字列, ブール, 比較演算子, ビット演算, 型注釈, capability, モジュール, ARC, 文の概念。

## オープン課題 (フェーズ2 前に決めたい)

- `i64` オーバーフロー方針の再検討 (Rust 流 / C 流 wrap / JS 流 f64 昇格)
- ファイル拡張子 `.il` の最終決定
- フェーズ2 で着手するもの:
  - 変数 (`let`) と文の概念
  - 関数定義 (Rust 風 `fn`)
  - **capability システムの骨組み** (ライブラリ単位の `net`/`file` 権限) ← サプライチェーン攻撃対策の核
- ネイティブコード化のロードマップ:
  - **第一候補は Cranelift** (Rust 製、`cargo build` だけで完結、JIT 起動が速い)
    - `ilang-mir` を挟んでから `ilang-codegen-cranelift` crate を追加
    - REPL の高速化と AOT ビルドの両方に利用
  - LLVM (inkwell) は後付けの最適化バックエンドとして検討 (リリースビルド用)
  - 採用理由: フェーズ初期は実装コストとビルド時間を優先。LLVM は最適化が必要になった段階で第二バックエンドとして追加できるよう、コード生成インタフェースを抽象化しておく
